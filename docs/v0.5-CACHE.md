# v0.5 — Cache local SQLite+FTS5 (runbook)

Runbook operativo del cache local del cliente `erickaldama-mail` — qué es, dónde vive, cómo se sincroniza, cómo
buscar, cómo diagnosticar problemas, y sus limitaciones conocidas. Complementa `CHANGELOG.md` (narrativa de
diseño) y `docs/superpowers/specs/2026-07-01-mail-v0.5-cache-design.md` (spec técnico completo).

---

## 1. Qué es

Un cache local **descartable** de los headers de correo (`sender`, `subject`, `date`, `s3Key`, claves de
DynamoDB), respaldado por SQLite + FTS5. Acelera `mail ls` (sirve desde disco en vez de golpear DynamoDB en
cada invocación) y habilita búsqueda full-text local (`mail search`, filtro `/` de la TUI).

**DynamoDB sigue siendo la única fuente de verdad.** El cache nunca es autoritativo — es un índice derivado.
Si se corrompe, se queda desactualizado, o simplemente se borra, se reconstruye solo en la siguiente
invocación que sincronice (`mail ls` o `mail search`). No hay pérdida de datos posible por perder el cache.

No cachea el cuerpo del mensaje (MIME) — eso sigue viviendo en S3 y se lee on-demand con `Reader.Open` en
`mail read`. El cache solo contiene metadata de listado/búsqueda.

## 2. Dónde vive

```
$XDG_CACHE_HOME/erickaldama-mail/index.sqlite      # si $XDG_CACHE_HOME está seteado
~/.cache/erickaldama-mail/index.sqlite             # fallback (macOS y Linux por igual — no usa
                                                    #   ~/Library/Caches, decisión explícita)
```

Resuelto por `cache.DefaultPath()` (`internal/cache/cache.go`).

**Permisos (verificados por test, no solo documentados):**
- Directorio contenedor: `0700`.
- Archivo `index.sqlite`: `0600` — forzado desde Go **antes** de que SQLite lo abra (`os.OpenFile(...,
  0o600)` + `os.Chmod` de refuerzo), porque SQLite no controla permisos de archivo por sí mismo; hereda el
  umask del proceso si no se interviene.
- Sidecars de modo WAL (`index.sqlite-wal`, `index.sqlite-shm`): también `0600` — contienen páginas de
  headers sin checkpoint, mismo nivel de sensibilidad que el archivo principal. Se aplican best-effort
  (toleran `ENOENT` si el sidecar aún no existe en el momento del `Open`).

**Es descartable.** Para reconstruirlo desde cero (por ejemplo, tras sospecha de corrupción, o simplemente
para forzar una resincronización completa):

```bash
rm ~/.cache/erickaldama-mail/index.sqlite*
```

El glob `*` cubre el archivo principal y los sidecars `-wal`/`-shm`. La siguiente invocación de `mail ls` o
`mail search` recrea el directorio, el archivo, el esquema, y sincroniza desde DynamoDB automáticamente —
no requiere ningún flag ni paso manual adicional.

## 3. Dependencia

`modernc.org/sqlite v1.53.0` — SQLite embebido 3.53.2, FTS5 compilado out-of-the-box (sin build tag), 100%
CGO-free (compila igual en darwin/arm64 y linux/amd64, sin toolchain de C). Driver registrado bajo el nombre
`"sqlite"` (no `"sqlite3"` — ese es el nombre que usa el driver con cgo, distinto paquete).

Es la primera dependencia de persistencia local del cliente. Trae 7 indirectas nuevas: `modernc.org/libc`,
`modernc.org/mathutil`, `modernc.org/memory`, `github.com/ncruces/go-strftime`, `github.com/dustin/go-humanize`,
`github.com/google/uuid`, `github.com/remyoudompheng/bigfft`. El módulo completo queda en ~45 dependencias
tras `go mod tidy`.

## 4. Esquema

```sql
CREATE TABLE headers (
  rowid     INTEGER PRIMARY KEY,   -- explícito: es el content_rowid natural del FTS
  s3Key     TEXT NOT NULL UNIQUE,  -- clave de conflicto del upsert
  pk        TEXT NOT NULL,         -- "mailbox#"+name
  sk        TEXT NOT NULL,         -- orden newest-first
  messageId TEXT,
  sender    TEXT,
  subject   TEXT,
  date      TEXT
);
CREATE INDEX idx_headers_mailbox ON headers(pk, sk DESC);

CREATE VIRTUAL TABLE headers_fts USING fts5(sender, subject, content='');  -- contentless

CREATE TABLE sync_state (
  mailbox   TEXT PRIMARY KEY,
  last_sk   TEXT,
  synced_at TEXT
);
```

`headers_fts` es **contentless** (`content=''`), no external-content, y **sin triggers**. Ver §5 para por qué
y cómo se mantiene sincronizado.

## 5. Flujo de sincronización (`Sync`)

Cada llamada a `mail ls` o `mail search` sincroniza antes de leer: trae una página de DynamoDB (tope
**fijo** `SyncPageLimit = 500`, independiente de cualquier flag `--count`/`--limit` que el usuario haya
pasado — esos solo acotan cuántas filas se **muestran**, no cuántas se sincronizan) y hace upsert en SQLite
dentro de una única transacción.

Por qué contentless-sin-triggers, y no external-content con triggers automáticos: el research previo dejó
sin verificar si un `INSERT ... ON CONFLICT DO UPDATE` dispara los triggers `AFTER INSERT`/`AFTER UPDATE`
que un esquema external-content necesitaría. Apostar a ese comportamiento no verificado arriesgaba corrupción
silenciosa del índice FTS si no disparaba como se esperaba. La alternativa elegida es contentless: Go
sincroniza el índice FTS **explícitamente**, sin depender de ningún trigger.

Esa elección exige un cuidado adicional, verificado empíricamente: una tabla FTS5 contentless **rechaza**
`DELETE FROM headers_fts WHERE rowid=?` (error: `cannot DELETE from contentless fts5 table`). Reemplazar la
entrada de un s3Key que ya existía requiere el comando especial `'delete'`, que además exige pasar los
valores **VIEJOS** actualmente indexados (no basta con el rowid). El orden dentro de la transacción es:

1. Leer `sender`/`subject` VIEJOS del `s3Key` (si la fila ya existía).
2. Si existía: `INSERT INTO headers_fts(headers_fts, rowid, sender, subject) VALUES('delete', ?, ?, ?)` con
   esos valores viejos — borra la entrada FTS existente sin usar `DELETE FROM`.
3. Upsert de `headers` (`ON CONFLICT(s3Key) DO UPDATE ... RETURNING rowid`) — captura el rowid (estable,
   se reusa en updates).
4. `INSERT INTO headers_fts(rowid, sender, subject)` con los valores nuevos.
5. Actualizar `sync_state` (mailbox → `last_sk`/`synced_at`).
6. Commit.

Re-sincronizar la misma página no duplica: el upsert es por `s3Key`, y el ciclo delete-viejo+insert-nuevo del
FTS es idempotente por rowid.

## 6. Cómo buscar

**CLI:**
```bash
mail search <query>
mail search --mailbox trabajo "factura pendiente"
mail search --json <query>        # JSON, igual que ls --json
```

`mail search` sincroniza el cache primero (best-effort — si falla, avisa por stderr y busca con lo que ya
había en disco), luego consulta `headers_fts MATCH` sobre `sender`+`subject`, ordenado por `rank`. A
diferencia de `mail ls`, `search` **no tiene fallback en vivo**: el FTS solo existe en el cache local, así
que si `cache.Open` falla, el comando aborta (no hay a dónde degradar).

**TUI:** presionar `/` abre el filtro. Cada tecla dispara `cache.Search(mailbox, query, 200)` y recarga la
lista con el resultado — mismo ranking FTS5 que `mail search`, sobre el histórico completo cacheado, no solo
los ítems ya cargados en pantalla. `Esc` cancela y restaura la lista completa (`cache.List`); Tab/Enter
confirman el filtro aplicado.

El texto del usuario pasa por `EscapeFTS` antes de llegar a `MATCH` — se envuelve entre comillas dobles
(duplicando cualquier `"` interna), lo que neutraliza `AND`/`OR`/`NOT`/`NEAR` como operadores booleanos de
FTS5 y los trata como texto literal. Ver limitación (e) más abajo por el trade-off que esto implica.

## 7. Troubleshooting

**`SQLITE_BUSY` / lock contention:**
El cache fuerza `SetMaxOpenConns(1)` + `PRAGMA busy_timeout=5000` + `journal_mode=WAL`. Si dos invocaciones
del CLI corren casi al mismo tiempo, la segunda espera (reintenta internamente) hasta 5 segundos antes de
fallar. En el uso normal (invocaciones interactivas, una a la vez) esto es invisible. Si aparece un error de
lock persistente más allá del timeout, lo más probable es un proceso colgado con el archivo abierto — revisar
procesos `mail`/`mail-tui` vivos (`pgrep -fl 'mail(-tui)?$'`) antes de asumir corrupción.

**Cache corrupto / comportamiento errático (resultados viejos, columnas vacías, error de SQLite al abrir):**
```bash
rm ~/.cache/erickaldama-mail/index.sqlite*
mail ls   # o `mail search <query>` — cualquiera de los dos recrea y resincroniza
```
No hay ningún estado que sobreviva al `rm` que pueda dejar el cliente en un estado peor que "recién
instalado" — el cache es puramente derivado de DynamoDB.

**Verificar permisos del archivo:**
```bash
stat -f %Lp ~/.cache/erickaldama-mail/index.sqlite   # macOS — debe imprimir 600
```

## 8. Limitaciones conocidas

Estas son decisiones de diseño documentadas, no bugs pendientes — cada una tiene un motivo y un mitigante.

**(a) El cache NO detecta correos borrados en DynamoDB — quedan huérfanos.**
`Sync` es puramente aditivo (upsert): trae headers de DynamoDB y los inserta/actualiza en SQLite, pero nunca
borra filas del cache que ya no existen en DynamoDB. Si un correo se elimina de DynamoDB por fuera del
cliente, su fila persiste en `headers`/`headers_fts` indefinidamente y seguirá apareciendo en `mail ls`/`mail
search` hasta que se fuerce una reconciliación manual: `rm ~/.cache/erickaldama-mail/index.sqlite*` seguido de
cualquier comando que sincronice. No hay barrido automático de huérfanos en v0.5.

**(b) `mail ls` sirve del cache; `tmux status` es LIVE — divergencia intencional.**
`mail ls` lee desde `cache.List` (con sync previo, pero fundamentalmente desde SQLite). `mail tmux status`
—el conteo que alimenta `status-right` de tmux— llama `reader.List` directamente contra DynamoDB, sin pasar
por el cache en absoluto. Es deliberado: el conteo del status bar debe ser el número autoritativo en todo
momento (es la señal de "¿tengo correo nuevo?"), mientras que el listado interactivo puede tolerar servir de
un cache que se resincroniza en cada invocación. Un usuario puede ver `mail ls` desactualizado por unos
segundos frente al conteo del status bar si llega correo nuevo justo entre sincronizaciones — es el trade-off
aceptado de tener listado rápido sin sacrificar la fuente autoritativa del conteo.

**(c) Dos invocaciones CLI concurrentes: la segunda espera hasta `busy_timeout` (5s), sin corrupción.**
`SetMaxOpenConns(1)` serializa el acceso dentro de cada proceso, y el modo WAL + `busy_timeout=5000` maneja el
caso de dos procesos distintos abriendo el mismo archivo casi simultáneamente (por ejemplo, `mail ls` en una
terminal y `mail-tui` en otra, lanzados en el mismo segundo). La segunda invocación espera hasta 5 segundos a
que la primera libere el lock; si la primera termina dentro de ese margen (el caso normal — una sync de ~500
filas es rápida), la segunda continúa sin error visible para el usuario. No hay riesgo de corrupción del
archivo por esta concurrencia — solo latencia adicional en el peor caso.

**(d) Los AI tools (`internal/aiassist`) usan substring, NO FTS5 — inconsistencia menor, diferida.**
La búsqueda que usan las tools del agente AI (`internal/aiassist/tools.go:69`,
`strings.Contains(strings.ToLower(h.Subject), strings.ToLower(q))`) sigue siendo substring case-insensitive
en memoria sobre los headers ya cargados — no consulta `cache.Search`/FTS5. Esto significa que la búsqueda que
ve el agente AI puede rankear/matchear distinto a `mail search` y al filtro `/` de la TUI para la misma
query. Es una inconsistencia conocida, de prioridad menor, diferida fuera de v0.5 — el agente AI no es la
superficie primaria de búsqueda del cliente.

**(e) La búsqueda FTS5 hace prefix match del ÚLTIMO token, no substring arbitrario ni fuzzy.**
`EscapeFTS` envuelve el query completo del usuario entre comillas dobles antes de pasarlo a `MATCH` — necesario
para neutralizar `AND`/`OR`/`NOT`/`NEAR` como operadores booleanos de FTS5 (sin el escapado, un query como
`factura OR pendiente` se interpretaría como expresión booleana, no como texto literal) — y añade un `*` **fuera**
de la comilla de cierre (`"factura pendiente"*`), que marca el último token como token de prefijo
(sqlite.org/fts5.html §3.3). Consecuencia práctica: escribir `transfer` **sí** matchea `transferencia` (prefijo del
último token). Matices vs el fuzzy de v0.4 (`sahilm/fuzzy`): (1) el prefix aplica solo al ÚLTIMO token —
`transf trans` no hace prefijo sobre `transf`, solo sobre el segundo `trans`; (2) NO es substring arbitrario en medio
de palabra (`ferencia` no matchea `transferencia`) ni tolera typos. Es prefijo por token, no fuzzy. El escapado sigue
neutralizando operadores/inyección; el `*` recupera la conveniencia del prefijo sin sacrificar esa seguridad.
