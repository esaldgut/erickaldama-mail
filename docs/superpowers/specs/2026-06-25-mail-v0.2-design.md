> Subproyecto #29. Mejoras al cliente de correo (SP-4) tras uso real el 2026-06-25.
> **Worktree:** `mail-v0.2` (rama `worktree-mail-v0.2`, base `875b3ee`). Go 1.26.4. Repo PÚBLICO Git Flow → cierre por PR.
> 2º fogueo de `subproject-delivery-canonical`.

---

## 1. Objetivo y motivación (hallazgos de uso real)

Tras instalar el cliente y usarlo, dos papercuts reales:
1. **El default `--mailbox inbox` nunca tiene correo.** La PK de DynamoDB es `mailbox#<dirección-destinataria>`
   (verificado en vivo: `mailbox#erick@erickaldama.com`, `mailbox#test@erickaldama.com`). No existe un buzón "inbox".
   El usuario envía a `erick@` y `mail ls` muestra 0 porque consulta `mailbox#inbox`.
2. **No hay CC/CCO.** El cliente solo soporta `From/To/Subject/Body` — ni en CLI (`mail send`), ni en el núcleo
   (`message.BuildOpts`), ni en el composer del TUI (que solo edita el cuerpo).

**El pipeline de recepción está OK** (verificado: 3 correos en S3 = 3 items DynamoDB, mismos s3Key, DLQ vacía).
El problema es solo del cliente, no del backend.

## 2. Ground truth verificado (no asumido)

| Hecho | Valor verificado |
|---|---|
| PK de DynamoDB | `mailbox#<destinatario>` (Query `PK = :pk` con `:pk = "mailbox#" + mailbox` en reader.go:55-57) |
| `enmime/v2` CC/BCC | `MailBuilder.CC(name, addr)`, `.BCC(name, addr)`, `.CCAddrs([]mail.Address)`, `.BCCAddrs(...)` existen |
| SES envelope para BCC | `ses.SendRawEmailInput.Destinations []string` — doc: "A list of destinations consisting of To:, CC:, and BCC:". Al pasarlo, SES entrega a esa lista; el header del raw solo muestra To+Cc → el BCC no se filtra |
| bubbles | `charmbracelet/bubbles v1.0.0` ya en go.mod (indirect vía bubbletea); pasar a directo con `go mod tidy` |
| Send actual | `Send(ctx, raw []byte) (string, error)` — cambia para aceptar la lista de destinations |
| Reader.List | `List(ctx, mailbox string, limit int32, start) ([]Header, cursor, error)` — Query por una PK |

## 3. Componentes

### 3.1 `internal/config/config.go` (nuevo) — config.toml
- Path: **XDG explícito, NO `os.UserConfigDir()`** (BLOCKER B-1b — en macOS `UserConfigDir()` devuelve
  `~/Library/Application Support`, NO `~/.config`, así que el path documentado nunca se hallaría):
  ```go
  func configPath() string {
      if xdg := os.Getenv("XDG_CONFIG_HOME"); xdg != "" {
          return filepath.Join(xdg, "erickaldama-mail", "config.toml")
      }
      home, _ := os.UserHomeDir()
      return filepath.Join(home, ".config", "erickaldama-mail", "config.toml")
  }
  ```
- Estructura:
```toml
mailboxes    = ["erick@erickaldama.com", "test@erickaldama.com"]
default_from = "erick@erickaldama.com"
read_profile = "mail-client-read"   # opcional
send_profile = "mail-sender"        # opcional
```
- **Dependencia NUEVA (B-1a):** `github.com/BurntSushi/toml` NO está en go.mod → el plan incluye `go get
  github.com/BurntSushi/toml` + `go mod tidy`/`verify` como paso explícito.
- `Load() (*Config, bool, error)`: parsea con BurntSushi. Archivo ausente → `(*Config zero, ok=false, nil)` (NO error).
  Archivo malformado → error con la ruta (BurntSushi reporta posición).
- Función pura `LoadFrom(path) (*Config, bool, error)` para testear sin tocar el HOME real.

### 3.2 `internal/message/build.go` — CC/BCC en el núcleo
- `BuildOpts` += `Cc, Bcc string` (coma-separados; helper `splitAddrs(s) []string` que trimea y descarta vacíos).
- En `Build()`: por cada Cc → `b.CC("", addr)`. **El Bcc NO se añade al builder** (no debe estar en el header MIME).
- `Build` retorna también la lista de destinations: cambiar a `Build(opt) (raw []byte, destinations []string, err error)`
  donde `destinations = To + Cc + Bcc` (todos, para el envelope SES). **Decisión:** Build es quien conoce todos los
  destinatarios, así que es el lugar natural para computar el envelope.

### 3.3 `internal/mailbox/sender.go` — envelope SES con BCC
- `Send(ctx, raw []byte, destinations []string) (string, error)` (firma nueva). Pasa `Destinations: destinations`
  en `SendRawEmailInput`. Si `destinations` está vacío (caller viejo), SES cae al header del raw (compat).
- El check `MaxRawBytes` y el manejo `ErrSandboxRecipient`/`DetectSandbox` se mantienen.

### 3.4 `internal/mailbox/reader.go` — sin cambios de firma
- `List` ya consulta una PK; el "consultar todos los mailboxes" se orquesta en el CLI (loop sobre `List` por cada
  mailbox del config), NO en el Reader. El Reader sigue siendo una operación atómica por-mailbox (responsabilidad única).
- **El `Header` debe exponer `SK`** (el atributo DynamoDB `SK = ts#<ISO8601 UTC>#<id>`) para que el CLI ordene el merge
  por él (B-2a). Verificar que el struct `Header` ya lo trae del Query (la proyección incluye SK) y que tiene json tag;
  si no se proyecta, añadirlo a la `ProjectionExpression`/struct.

### 3.5 `cmd/mail/main.go` — CLI
- **`ls` sin `--mailbox`**: si hay config con `mailboxes`, hacer un `List(mailbox, limit)` por **cada** mailbox
  (limit COMPLETO por mailbox, NO repartido — B-2b: repartir perdería correos), mergear todos, **ordenar por `SK`
  descendente** (NO por `Date` — B-2a: el `Date` RFC1123Z `"Thu, 25 Jun..."` ordena lexicográficamente MAL; el `SK`
  `ts#2026-06-25T07:23:35Z#<id>` es ISO8601 UTC y SÍ ordena correcto), truncar el merge a `limit`. Si un mailbox falla,
  reportar ESE error a stderr y seguir con los demás (no abortar todo).
- **`ls` sin config Y sin `--mailbox` → ERROR claro con exit≠0** (B-5, decisión del usuario): `"no hay config; crea
  ~/.config/erickaldama-mail/config.toml con tus mailboxes, o usa --mailbox <dirección>"`. NO degradar a `inbox`
  (que siempre da 0 — es el papercut que este subproyecto resuelve).
- **`send`/`reply`**: flags `--cc`, `--bcc`. `--from` cae a `config.default_from` si está y no se pasó. `Build` retorna
  `(raw, destinations, err)`; `Send(ctx, raw, destinations)`. **A-4: 4 call-sites a actualizar en este archivo** —
  Build en `send` (~159) y `reply` (~241); Send en `send` (~174) y `reply` (~258).
- **`--cc`/`--bcc` asumen direcciones PELADAS** (`--cc a@x,b@y`). El formato `"Nombre <addr>"` NO se soporta en v0.2
  (A-5b: `splitAddrs` lo pasaría mal-formado a enmime → Cc inválido). Nota en --help. Reply-all sí pre-rellena
  direcciones peladas (ver §3.6).
- Las flags persistentes `--read-profile`/`--send-profile` caen al config si está y no se pasan en CLI.

### 3.6 `cmd/mail-tui/model.go` — composer multi-campo + reply-all
- Reemplazar el `composeDraft string` único por un struct `composer` con campos `bubbles/textinput` para
  To/Cc/Bcc/Subject + el body vía $EDITOR (el flujo `editorCmd`/`tea.ExecProcess` actual se mantiene).
- **Integración de `textinput.Model` (B-3b, no trivial):** el modelo padre enruta `tea.KeyMsg` al input activo
  (`inputs[active].Update(msg)`), llama `textinput.Blink` como `tea.Cmd` en Init, y gestiona focus/blur al navegar.
  Descomponer en pasos en el plan (no "reescribir el composer" en una sola tarea).
- Navegación Tab/Shift-Tab entre los 4 textinputs; Ctrl-E abre $EDITOR para el body; Ctrl-S → confirm `y/n` (el gate
  de envío actual se preserva — audit H-2 de SP-4).
- **Actualizar los 3 tests que el refactor rompe (B-3a):** `TestComposerSendRequiresConfirmation` (model_test.go:46),
  `TestSentMsgClearsConfirmState` (63, asserta `composeDraft != ""` → cambiar al nuevo state), `TestReplyDraftPrePopulates`
  (73, compara el string de `replyDraft` → adaptar al nuevo contrato). No romperlos silenciosamente.
- `View()` del composer renderiza los 4 campos + un preview del body.

### 3.7 `internal/message/parse.go` — Cc/To para reply-all (decisión del usuario: reply-all completo)
- Añadir `Cc string` y `To string` a `Parsed`; capturarlos en `Parse` vía `env.GetHeader("Cc")`/`env.GetHeader("To")`
  (enmime los expone). Test del parser actualizado (un fixture con Cc).
- **Reply-all:** al responder, pre-rellenar el campo Cc del composer con los destinatarios `To`+`Cc` del original
  MENOS la propia identidad (`--from`/`default_from`), direcciones peladas. El flujo 'r' del TUI necesita el mensaje
  PARSEADO (hoy solo tiene el `Header` de DynamoDB) → abrir+parsear el mensaje seleccionado antes de poblar el composer.
- El CLI `reply` también: pre-rellena `--cc` con los originales si el usuario no pasa `--cc` explícito (override si lo pasa).

## 4. Manejo de errores

| Caso | Manejo |
|---|---|
| config.toml ausente | NO error: degrada al default actual + hint stderr |
| config.toml malformado | error claro con la ruta + línea (BurntSushi reporta posición) |
| `--cc`/`--bcc` con dirección inválida | enmime/SES la rechaza; surface el error tipado (no string-match) |
| BCC en el header (regresión) | **test que verifica que el raw MIME NO contiene "Bcc:"** — el BCC solo en Destinations |
| ls multi-mailbox, un mailbox falla | reportar el error de ESE mailbox a stderr, seguir con los demás (no abortar todo) |

## 5. Seguridad y privacidad

- **BCC nunca en el header MIME** — solo en `SendRawEmailInput.Destinations`. Es el invariante de privacidad central;
  un test lo asegura. Si el BCC se filtrara al header, todos los destinatarios verían las copias ocultas.
- **Menor privilegio intacto**: el ls multi-mailbox usa Query (no Scan) → NO requiere ampliar la policy de
  `mail-client-read`. Cero cambio de IAM.
- **config.toml**: no contiene secretos (solo direcciones + nombres de profile). Los profiles AWS siguen en `~/.aws`.
- **Repo público**: gate NDA sobre todo el output. El config.toml de ejemplo en docs usa direcciones publicables
  (`erick@erickaldama.com` es publicable).

## 7. Definition of Done

1. `go build ./... && go test -count=1 ./...` verde.
2. `mail ls` (con config) lista correos de todos los mailboxes; `mail ls --mailbox X` sigue funcionando.
3. `mail send --cc a@x --bcc b@y` → el destinatario recibe; el BCC recibe; el header NO muestra el Bcc (test + smoke real).
4. El composer del TUI tiene campos To/Cc/Bcc/Subject navegables + body vía $EDITOR.
5. config.toml de ejemplo documentado en el runbook (docs/SP-4-DEPLOY.md o un v0.2 nuevo).
6. PR a develop con CI verde. NDA limpio.

## 8. Non-goals (v0.2)

- Sin Scan ni GSI (el multi-mailbox es Query-por-config; un catch-all real "todos los destinatarios sin listarlos"
  necesitaría Scan/GSI + ampliar IAM → v0.3 si se quiere).
- Sin adjuntos en el composer del TUI (el CLI ya los tiene vía BuildOpts.Attachments; el TUI queda para después).
- Sin libreta de direcciones / autocompletado de destinatarios.
- Sin HTML compose (el body sigue siendo texto plano vía $EDITOR).
