## Decisiones de diseño (cerradas en el brainstorm)

### D1 — Layout: 2 paneles simultáneos + focus alternable con Tab (estilo mutt/aerc)
Lista (izq ~35%) y reader (der ~65%) visibles a la vez vía `lipgloss.JoinHorizontal(lipgloss.Top, listPane, readerPane)`.
Un campo `focus` (`focusList`/`focusReader`) enruta las teclas. **Tab** alterna. El panel con foco lleva border
resaltado (`RoundedBorder` + `BorderForeground(AdaptiveColor{Light,Dark})`); el otro, border tenue. Cierra H1 de raíz:
ves lista + contenido juntos. (Descartadas: pantallas-alternas — no cierra H1; reader-preview — añade un modo extra.)

### D2 — Los 4 modos del modelo (reemplazan los 3 viewState actuales)
| Modo | Pantalla | Teclas |
|---|---|---|
| `modeBrowse` | 2 paneles (list + reader) | Tab alterna focus; teclas → panel con foco |
| `modeFilter` | 2 paneles, list filtrando | **TODAS** las teclas → `list` (regla D3) |
| `modeComposer` | overlay full-screen | `handleComposerKey` (de v0.2, intacto) |
| `modeConfirm` | overlay de confirmación | y/n |

### D3 — Regla de oro del focus (cierra el NO VERIFICADO #1 de Fase 0)
Antes de enrutar cualquier `KeyMsg`: si `m.list.SettingFilter()` es true → la tecla va **entera** a `m.list.Update()`
(el usuario está tipeando el filtro). Evita que Tab/Enter/letras se las robe el layout durante el filtrado. Los 3
estados: `Unfiltered` (navega normal), `Filtering` (robar teclas), `FilterApplied` (filtro fijo, el padre vuelve a
navegar entre paneles).

**Colisión Tab↔filtro (audit B-2, CRIT, verificado ejecutando).** En bubbles v1.0.0 (`list/keys.go:75`), la acción
`AcceptWhileFiltering` está bound a `tab`/`shift+tab` además de `enter`. Por tanto, mientras `SettingFilter()` es true,
**Tab NO alterna paneles** — aplica el filtro (pasa a `FilterApplied`) y el foco se queda en la lista. Esto es
CONSISTENTE con D3 (durante el filtro, todas las teclas son de la lista) y es el comportamiento correcto: el usuario
teclea el filtro, Tab/Enter lo fija, y *recién entonces* Tab vuelve a alternar paneles. El spec lo documenta para que
el smoke de Fase 5 ("Tab alterna el foco") se pruebe FUERA de un filtro activo, no dentro. La regla de Tab en el
ciclo Update (paso 3) se condiciona a `!m.list.SettingFilter()`.

### D4 — Composer = overlay pantalla-completa (4º modo, separado del layout de lectura)
Al presionar `c` (componer) o `r` (responder), el composer toma toda la pantalla como overlay; al enviar o cancelar
(Esc) vuelve a los 2 paneles. El `Tab` interno del composer (navega campos To/Cc/Bcc/Subject) **NO choca** con el Tab
de focus entre paneles porque solo un modo está activo a la vez. El composer de v0.2 queda casi intacto, solo se
extrae a su propio archivo.

### D5 — Lista: Title=Subject, Desc=From+fecha, FilterValue=From+Subject
Cada `mailbox.Header` se vuelve un `messageItem` que implementa `list.DefaultItem`:
- `Title()` → `Subject`, **con fallback `"(sin asunto)"` si `Subject==""`** (audit B-6: `Header.Subject` es
  `omitempty` → puede venir vacío de DynamoDB → Title en blanco → item invisible en el delegate). El fallback evita la
  línea fantasma.
- `Description()` → `From + " · " + fechaCorta` (contexto sin ruido)
- `FilterValue()` → `From + " " + Subject` (el filtro `/` busca por quién o por tema)

**`messageItem` DEBE portar `S3Key` y `MessageID`** (audit B-7, MAYOR): el refactor borra `m.headers[]`, pero 5
call-sites de reply/load indexan hoy `m.headers[m.selected].S3Key` (`model.go:242,251,309,359,367`). Esos paths pasan
a leer `m.list.SelectedItem().(messageItem).S3Key`. Sin `S3Key` en el item, el reply pierde el correo origen. El
`messageItem` envuelve el `mailbox.Header` completo (no solo los 3 campos de display) para no perder datos.

Usa el `DefaultDelegate` verificado en Fase 0 (cero riesgo nuevo). El delegate custom de 1-línea (densidad) se difiere.

### D6 — Spinner async con generation-counter (NO un bool simple) + anti tick-storm
**El bool simple no sirve (audit A-4, CRIT, race verificado ejecutando).** Con un único `loading bool` y un
`bodyLoadedMsg` sin request-ID, dos cargas que se solapan producen un bug: selecciono A (carga lenta), muevo a B
(carga rápida), B se muestra, A vuelve tarde y **sobrescribe el viewport** → `selected=="B"` pero el reader muestra
el body de A. Y la primera op que vuelve apaga el spinner aunque otra siga en vuelo. D6 (loadBody en cada movimiento,
no solo Enter como v0.3) ENSANCHA esta ventana de carrera.

**Mecanismo correcto:**
- El `model` lleva `loadGen int` (contador monótono). Cada loadBody lo incrementa y captura el valor; el
  `bodyLoadedMsg` **transporta `gen int`**. Al recibirlo, si `msg.gen != m.loadGen` → **se descarta** (es una
  respuesta stale de un correo ya no seleccionado). Solo la respuesta del gen vigente puebla el viewport.
- Contador de operaciones en vuelo `inflight int` (no un bool): cada async lo incrementa, cada retorno lo decrementa;
  el spinner se muestra mientras `inflight > 0`. Así el spinner no desaparece con trabajo pendiente.
- **Debounce del loadBody (audit B-1/C3):** al mover el cursor con j/k rápido, NO disparar una carga S3 por tecla.
  Disparar la carga tras un `tea.Tick` corto (≈150 ms) de quietud, o solo cuando la selección se estabiliza. Evita el
  N+1 de requests + spinner-storm al recorrer la lista.

**Las 3 operaciones async** (cada una incrementa `inflight`, su retorno lo decrementa):

| Disparo | Cmd async | Mensaje retorno (lleva `gen` donde aplique) |
|---|---|---|
| Selección estabilizada (debounce) | carga body S3 + parse | `bodyLoadedMsg{parsed, gen}` — descarta si `gen` stale |
| Tecla `i` (reader) | render chafa imágenes | `imagesRenderedMsg{blobs}` |
| Ctrl-S (composer) | envío SES | `sentMsg{messageID}` |

**Anti tick-storm (audit A-3, MAYOR):** el `spinner.Tick` con `tag==0` siempre se acepta, así que arrancar el tick en
cada una de las 3 ops inicia cadenas `tea.Tick` paralelas (frames 2×, timers duplicados durante el solape). Regla:
arrancar `m.spinner.Tick` SOLO en la transición `inflight 0→1`, y enrutar `m.spinner.Update` SOLO en el case
`spinner.TickMsg` (no en el flujo general de Update).

### D7 — Mouse + WithMouseCellMotion (con enrutado explícito de MouseMsg)
`tea.NewProgram(m, tea.WithAltScreen(), tea.WithMouseCellMotion())` — se **añade** `WithMouseCellMotion()` (hoy falta;
el viewport lo necesita para scroll de rueda) + `vp.MouseWheelEnabled = true`. Solo scroll de rueda; no drag/click en items (YAGNI).

**El ciclo Update DEBE enrutar `tea.MouseMsg` al viewport (audit A-1b, MAYOR).** `viewport.Update` es el único lugar
que reacciona a la rueda, y solo lo hace si le llega el `MouseMsg`. Si el Update solo maneja `KeyMsg` (como sugería el
borrador del ciclo), D7 es código muerto. Fix: un `case tea.MouseMsg` explícito en el switch de tipo de `Update` que
reenvíe a `m.viewport.Update(msg)` cuando el reader es relevante.

### D8 — Cierra la deuda v0.3 (WB-4 + WB-2)
- **WB-4** (migrar scrollOffset+images al viewport): se cierra solo — el viewport REEMPLAZA scrollOffset, y el render
  de imágenes (tecla `i`) se integra como contenido del viewport (re-`SetContent` con las imágenes).
- **WB-2** (tests del TUI para R/imagesRenderedMsg): se cierra con los tests del nuevo flujo (ver Testing).

## Flujo de datos — el ciclo Update

```
Update(msg):                                    // switch por TIPO de msg (no solo KeyMsg)
  case tea.WindowSizeMsg → recalcular layout (ver Resize); vpReady=true
  case tea.MouseMsg      → m.viewport.Update(msg)    (A-1b: la rueda SOLO scrollea si llega aquí)
  case spinner.TickMsg   → m.spinner.Update(msg)     (A-3: spinner.Update SOLO aquí, no en flujo general)
  case bodyLoadedMsg     → si msg.gen==m.loadGen: sanitizar→RenderRich→viewport.SetContent; inflight--
                           si msg.gen!=m.loadGen: DESCARTAR (stale, A-4)
  case imagesRenderedMsg → viewport.SetContent(sanitizedBody + blobs); inflight--   (SAN: ver invariante)
  case sentMsg / errMsg  → inflight-- ; status
  case tea.KeyMsg:
    1. modeComposer/modeConfirm?      → handleComposerKey / y-n        (overlay manda)
    2. m.list.SettingFilter()?        → TODO → m.list.Update()         (D3: no robar teclas; Tab aquí APLICA filtro, B-2)
    3. teclas globales (q,c,r,s,a):   → switch
       Tab (solo si !SettingFilter()) → alterna m.focus                (B-2: Tab no alterna durante filtro)
    4. según m.focus:
         focusList   → prevKey := selectedKey(m); m.list.Update(msg)
                       newKey := selectedKey(m)   // selectedKey = SelectedItem().(messageItem).S3Key, "" si nil/filtering
                       si newKey != prevKey && newKey != "" && !SettingFilter() → scheduleLoadBody (debounce, B-1)
         focusReader → m.viewport.Update(msg)                          (j/k/pgup internos)
```

**Trigger de loadBody por identidad estable, no por `Index()` (audit B-1, CRIT).** bubbles/list NO emite "selección
cambió". La señal intuitiva `list.Index()` está ENVENENADA: abrir el filtro con `/` llama `GoToStart()` → `Index()`
salta a 0 sin navegación del usuario → cargaría el correo #0 espurio. La señal correcta es el `S3Key` del
`SelectedItem()` (identidad estable), comparado antes/después del Update, y **solo** cuando `!SettingFilter()`.
`selectedKey()` hace type assertion **comma-ok** sobre `SelectedItem()` → devuelve `""` si la lista está vacía
(audit B-3, evita el panic `interface conversion: list.Item is nil`).

**Reader load (`bodyLoadedMsg`):** sanitizar (P-5, intacto v0.3) → `RenderRich(clean, readerWidth)` →
`viewport.SetContent(body)` → `viewport.GotoTop()` → `inflight--`.

### Aritmético del layout (audit C1/C2, ambos CRIT — el desborde se midió ejecutando)
**El reparto 35/65 ingenuo desborda en AMBOS ejes.** Medido con `lipgloss.Width(JoinHorizontal(...))` contra dimensiones
reales: en 80 cols el join produce líneas de **82** (el `RoundedBorder` de la lista, +2 cols, nunca se descontó — el
borrador solo restaba el frame del reader). Y a `termHeight−1` el border top+bottom (+2 filas) empuja el footer fuera.
El cálculo CORRECTO descuenta los frames ANTES de repartir:

```
const frameW = 2  // RoundedBorder lateral por panel: GetFrameSize() = (2, 2)
const frameH = 2  // RoundedBorder top+bottom por panel
footerH := 1      // la línea de footer global

availW := termWidth - 2*frameW            // descontar el border de AMBOS paneles
listInnerW  := availW * 35 / 100
readerInnerW := availW - listInnerW       // el resto al reader (evita perder cols por redondeo entero)
panelInnerH := termHeight - frameH - footerH   // border vertical + footer

m.list.SetSize(listInnerW, panelInnerH)
m.viewport.Width  = readerInnerW
m.viewport.Height = panelInnerH
// readerWidth para RenderRich == readerInnerW (el border va POR FUERA, el probe confirmó que el viewport rellena al ancho)
```
Verificado que el corregido CABE exacto para termWidth ∈ {80,100,120,132}: `lipgloss.Width(join) == termWidth`.

**Resize (`WindowSizeMsg`):** aplicar el aritmético de arriba → `list.SetSize` → `viewport.Width/Height = ...` →
re-`RenderRich` al nuevo `readerInnerW`. Guard `vpReady`: no renderizar paneles hasta el primer resize (NO VERIFICADO
#3 de Fase 0). **Debounce (audit C3, MAYOR):** arrastrar el borde dispara docenas de `WindowSizeMsg`/s, cada una
re-corriendo htmltomd+glamour (caro). Coalescer: re-`RenderRich` tras un `tea.Tick` corto de quietud, no en cada
`WindowSizeMsg` (el `SetSize`/`Width` de list/viewport sí es barato y va inmediato; solo el `RenderRich` se debounce-a).

## Invariante de seguridad (audit B verificó el chain ejecutando — se preserva con 1 condición DURA)
Los SAN-1..5 de v0.3 se mantienen: el sanitizado ocurre en `bodyLoadedMsg` ANTES de `SetContent`, exactamente donde
estaba antes de `RenderRich` en v0.3. El audit B trazó el chain real (`rawHTML → SanitizeHTML → san.HTML → RenderRich
→ body`) ejecutándolo y confirmó: el path `R` re-sanitiza desde `rawHTML` y solo asigna el body post-RenderRich (el
tracker remoto se neutraliza a `[imagen remota bloqueada]`, nunca llega crudo); `imagesRenderedMsg` usa bytes `cid:`
de adjuntos decodificados (NO un fetch de URL).

**Condición DURA que el spec impone (audit B, agujero de redacción cerrado):** todo `viewport.SetContent` recibe el
**`body` POST-RenderRich sanitizado**, jamás `rawHTML`. En particular `imagesRenderedMsg` hace
`SetContent(sanitizedBody + blobs)` — re-usa el body ya sanitizado guardado, **NUNCA re-deriva de `rawHTML`**. El
whole-branch review confirma con grep que ningún `SetContent` recibe `rawHTML` o `currentParsed.TextHTML` crudo.
(NDA: sin direcciones reales en tests; fixtures sintéticos.)
