## v0.3 — Render rico (el alcance a ejecutar AHORA)

### Objetivo
El reader del correo muestra el body con **imágenes** y **HTML sanitizado**, en terminal, **sin filtrar privacidad**
(tracking pixels bloqueados) y **sin fallar** si faltan binarios externos.

### Decisiones de diseño (cerradas en brainstorm)

**D1 — Terminal primario, web secundario.** El render rico vive en la terminal (coherente con el stack nvim/tmux). El
modo web (`mail read --web`, servidor local + browser) se **difiere a v0.3.1**. v0.3 = solo terminal.

**D2 — Imágenes on-demand (no inline-siempre).** El reader muestra el body en markdown (glamour) por default. Una
**tecla explícita** (p.ej. `i`) renderiza las imágenes con chafa en ese momento. Razón: tmux BORRA las imágenes al
cambiar de pane (limitación dura verificada — findings doc 20 §A-2); on-demand evita esa fragilidad — la imagen es
una acción consciente, re-disparable, no parte del scroll.

**D3 — Bloquear imágenes remotas por default + flag consciente.** Los tracking pixels (img src=http remotos) filtran
IP/apertura — vector #1 (findings doc 20 §B-2). Por default: solo se renderizan imágenes **cid:** (inline, ya en el
MIME, sin red). Las remotas → placeholder `[imagen remota bloqueada]`. Un flag (`--load-remote`) o tecla las carga si
el usuario confía (patrón Apple Mail "Block All Remote Content"). Máxima privacidad por default, escape consciente.

**D4 — Sanitización SIEMPRE primero (orden obligatorio).** Antes de cualquier render:
```
1. Parsear (enmime/v2 — ya está)
2. Bloquear imágenes remotas (src=http → placeholder); permitir solo cid:
3. Sanitizar (bluemonday.UGCPolicy + AllowStandardURLs; bloquear data: en src/href)
4. CSS: bloquear <style> y style= completos
```
Bloquear SIEMPRE: `<script>`, `on*`, `javascript:`, `<iframe>`, `<object>`, `<embed>`, remotas, `<style>`/`style=`, `data:`.

**D5 — Degradar con gracia si faltan deps externas.** chafa es un binario externo (no viene con el cliente Go). Al
arrancar, detectar si chafa existe en PATH. Si está: render de imágenes. Si NO: mensaje claro `[chafa no instalado —
brew install chafa para ver imágenes]` y el cliente sigue funcionando con texto. **El cliente NUNCA falla por una dep
externa ausente.** Documentar en README.

**D6 — Fix del ancho de glamour (adelantado de v0.4).** Hoy `internal/message/render.go:27` usa `glamour.Render(md,
"auto")` que tiene **ancho fijo 80** → se ve mal en terminales anchos/angostos. Migrar a `glamour.NewTermRenderer(
glamour.WithAutoStyle(), glamour.WithWordWrap(width))`. Guardar el markdown CRUDO (no el ANSI) para re-renderizar al
ancho correcto. Es el único cambio funcional necesario en render.go, gran mejora del reader donde vive el render rico.

### Arquitectura v0.3 (componentes)

```
internal/message/
  parse.go     (ya) — Parsed con TextHTML
  render.go    (modif D6) — RenderRich con ancho dinámico (NewTermRenderer + WithWordWrap)
  sanitize.go  (NUEVO) — SanitizeHTML(html) → html limpio; bloquea remotas, devuelve lista de cid: + remotas-bloqueadas
internal/render/        (NUEVO paquete)
  images.go    — detecta chafa en PATH; RenderImage(cidData []byte, width int) → ANSI/Unicode via chafa shell-out;
                 degrada con mensaje si chafa ausente
cmd/mail/main.go        — flag --load-remote en read; el reader CLI muestra placeholders de remotas
cmd/mail-tui/model.go   — tecla 'i' = render imágenes on-demand; tecla para --load-remote; placeholders
```

### Invariantes de seguridad (a testear)
- **SAN-1:** tras SanitizeHTML, el HTML NO contiene `<script>`, `on*=`, `javascript:`, `<iframe>`, `<style>`, `style=`.
- **SAN-2:** las imágenes remotas (src=http) NO se cargan por default → NO hay request de red al renderizar (el test
  verifica que con --load-remote=false, ninguna URL remota se resuelve; solo cid: se procesa).
- **SAN-3:** chafa se invoca con `exec.Command` argv-slice (NUNCA `sh -c` con interpolación — anti injection). El
  contenido de la imagen va por stdin o archivo temp con permisos 0600, no por la línea de comando.
- **SAN-4:** degradación: chafa ausente → el cliente NO crashea; mensaje claro + sigue con texto.

## v0.4 — TUI estética (visión, NO se ejecuta en v0.3)

- **Reader → viewport.Model** (bubbles v1: `viewport.New(w,h)`, campos `.Width/.Height` directos, `SetContent`,
  `ScrollPercent()`). Header con borde lipgloss (RoundedBorder `├`/`┤` + relleno `─`), footer con `%`. Mide altura con
  `lipgloss.Height(header/footer)`, guard `vpReady`, re-renderiza glamour al ancho en cada resize. Elimina scrollOffset
  casero (model.go:73,96,259). **OJO: el ejemplo oficial del pager ya usa bubbles v2 — usar API v1, no copiar verbatim.**
- **Layout 2-paneles:** `lipgloss.JoinHorizontal(Top, listPane, readerPane)` con bordes + AdaptiveColor{Light,Dark};
  foco marcado con BorderForeground(activeColor).
- **Lista → bubbles/list** (fuzzy filter + paginación nativos) — la búsqueda casi gratis.
- **Async + spinner** (bubbles/spinner, NO briandowns) durante I/O.

## v0.5 — Cache local (visión, NO se ejecuta en v0.3)

- **modernc.org/sqlite v1.53.0** (puro-Go, sin CGO — binario distribuible) + **FTS5** (default en modernc).
- Índice de **headers solamente** (from/subject/date/s3Key/mailbox — NUNCA el body → reduce superficie NDA) en
  `~/.cache/erickaldama-mail/index.sqlite` (os.UserCacheDir), permisos 0600 + sidecars -wal/-shm.
- **Cache descartable** reconstruible 100% desde DynamoDB+S3. Sync: `ON CONFLICT(s3Key) DO UPDATE` idempotente +
  cursor del SK más reciente. WAL para que la TUI lea sin bloquear el sync. `mail ls`/búsqueda instantáneos offline.

## Definition of Done (v0.3)
1. `go build ./... && go test -count=1 ./...` verde.
2. SAN-1..SAN-4 testeados (sanitización, no-red-en-remotas, argv-slice, degradación).
3. `mail read` muestra el body con ancho correcto (fix D6) + placeholders de imágenes remotas bloqueadas.
4. La tecla de imágenes on-demand (TUI) renderiza cid: con chafa (o degrada con mensaje claro si falta chafa).
5. `--load-remote` carga las remotas con warning consciente.
6. Gate NDA sobre el diff. Smoke: render de un correo real con imagen (gate humano para correos reales — NDA).
7. README documenta: deps opcionales (chafa), la política de privacidad (remotas bloqueadas), el flujo de imágenes.
8. PR a develop, CI verde.

## Non-goals (v0.3)

- Sin w3m/elinks (elinks ejecuta JS — descartado; w3m diferido, evaluar en v0.3.1 con el web). Sin chromedp.
