## 1. Objetivo y no-objetivos

**Objetivo:** un cache local descartable que (a) acelere `mail ls` sirviendo headers desde SQLite en vez de golpear
DynamoDB en cada invocación, (b) habilite búsqueda full-text local sobre `sender`/`subject` vía FTS5 — expuesta como
subcomando CLI **y** como el filtro de la TUI, y (c) resuelva dos puntos de dolor de display del listado:
**la falta de hora** (hoy solo fecha → no se distinguen correos del mismo día) y **la ausencia del s3Key**
(hoy hay que re-listar/inspeccionar para conseguir el identificador que `mail read <s3Key>` necesita para abrir links).

**No-objetivos (YAGNI):**
- NO es fuente de verdad — DynamoDB lo es. El cache se reconstruye si falta/corrompe.
- NO cachea el cuerpo MIME (vive en S3, se sigue leyendo on-demand con `Reader.Open`).
- NO implementa copia del s3Key en el cliente — el usuario copia con el **copy mode de tmux** (el cliente solo lo
  muestra visible/seleccionable). El cliente NO añade clipboard ni OSC-52.
- NO paginación fina expuesta — los call-sites actuales traen la página completa (limit alto, cursor descartado).

## 3. Arquitectura — cache como decorador del Reader

Nuevo paquete `internal/cache`. Expone la misma forma que hoy consumen los call-sites (`[]mailbox.Header` newest-first
por mailbox), sirviendo desde SQLite y refrescando desde DynamoDB por detrás.

```
internal/cache/
  cache.go    — Cache: Open/Close SQLite, PRAGMAs por conexión, path 0600
  schema.go   — DDL (headers + headers_fts contentless) + init idempotente
  sync.go     — Sync(ctx, r *mailbox.Reader, mailbox string): DynamoDB → SQLite (upsert + FTS explícito)
  query.go    — List(mailbox, limit) []Header ; Search(mailbox, q, limit) []Header (FTS5)
```

**Interfaz pública (borrador):**
```go
type Cache struct { db *sql.DB }

func Open(path string) (*Cache, error)                 // crea dir 0700, archivo 0600, aplica PRAGMAs, init schema
func (c *Cache) Close() error
func (c *Cache) Sync(ctx context.Context, r *mailbox.Reader, mailbox string) (int, error)  // devuelve #filas upsert
func (c *Cache) List(mailbox string, limit int) ([]mailbox.Header, error)                   // desde SQLite, newest-first
func (c *Cache) Search(mailbox, query string, limit int) ([]mailbox.Header, error)          // FTS5 MATCH, ORDER BY rank
```

**Integración en los call-sites:** el flujo `ls` pasa a: `Open cache → Sync(reader, mb) → cache.List(mb, limit)`.
Si `Open`/`Sync` fallan (cache no disponible), **fallback transparente** a `reader.List` directo (el cache nunca
bloquea el flujo — es una optimización). El fallback se decide en el call-site, no dentro del cache.

**Superficies de esta versión (4 frentes):**
1. **Cache** (`internal/cache`) — SQLite + FTS5, sync, list, search (§4-§7).
2. **CLI display** (`cmd/mail/main.go renderList`) — hora + s3Key en la salida de `mail ls` (§6b).
3. **CLI search** (`cmd/mail search`) — subcomando nuevo (§6a).
4. **TUI** (`cmd/mail-tui/listitem.go` + `model.go`) — 3ª línea con s3Key (dim), fecha+hora, y el filtro `/`
   pasa de fuzzy-en-memoria a FTS5-sobre-cache (§6c).

## 4. Esquema — resuelve los 2 hallazgos críticos del research

**Contentless FTS5 + sync explícito en Go** (sin triggers, sin external-content):

```sql
CREATE TABLE IF NOT EXISTS headers (
  rowid     INTEGER PRIMARY KEY,        -- rowid explícito → content_rowid natural (resuelve crítico #2)
  s3Key     TEXT NOT NULL UNIQUE,       -- clave de conflicto del upsert (requisito UNIQUE de ON CONFLICT)
  pk        TEXT NOT NULL,              -- "mailbox#"+name
  sk        TEXT NOT NULL,              -- clave de orden (newest-first)
  messageId TEXT,
  sender    TEXT,                       -- Header.From
  subject   TEXT,
  date      TEXT
);
CREATE INDEX IF NOT EXISTS idx_headers_mailbox ON headers(pk, sk DESC);

CREATE VIRTUAL TABLE IF NOT EXISTS headers_fts USING fts5(sender, subject, content='');

CREATE TABLE IF NOT EXISTS sync_state (
  mailbox   TEXT PRIMARY KEY,
  last_sk   TEXT,                       -- el sk más reciente visto (para el incremental)
  synced_at TEXT
);
```

**Por qué contentless (no external-content):** el research dejó NO VERIFICADO si un upsert dispara el trigger
`AFTER INSERT` o `AFTER UPDATE`. Sin triggers, ese firing deja de importar — Go sincroniza el FTS explícitamente.
Elimina el **crítico #1** (corrupción silenciosa del índice) por construcción. Contentless devuelve NULL en columnas
salvo rowid, pero **eso es aceptable**: recuperamos `sender/subject/...` de `headers` (join por rowid), no del FTS.

> **Divergencia research→spec anotada (audit m-6):** el research §2 recomendó external-content; este spec la invierte
> a contentless para eludir el NO VERIFICADO #1 (triggers). Esa inversión reabrió una verificación que la auditoría
> del plan hizo: **contentless NO soporta `DELETE FROM ... WHERE rowid`** (verificado: "cannot DELETE from contentless
> fts5 table"). El borrado del índice usa el comando especial `INSERT INTO headers_fts(headers_fts,'delete',rowid,
> sender,subject) VALUES('delete',...)` con los valores VIEJOS actualmente indexados (§6.3 del research). Ver §5.

## 5. Sync incremental (`sync.go`)

Por cada mailbox, dentro de **una transacción**:
1. `hs, _, err := r.List(ctx, mailbox, limit, nil)` — trae la página desde DynamoDB (como hoy).
2. Por cada `Header h`:
   - `INSERT INTO headers (s3Key,pk,sk,messageId,sender,subject,date) VALUES (?,...) ON CONFLICT(s3Key) DO UPDATE SET
     pk=excluded.pk, sk=excluded.sk, messageId=excluded.messageId, sender=excluded.sender, subject=excluded.subject,
     date=excluded.date RETURNING rowid` → capturar el `rowid`. `RETURNING` está soportado desde SQLite 3.35.0
     (driver embebe 3.53.2) y **funciona con UPSERT** ("A RETURNING clause for an UPSERT reports both inserted and
     updated rows" — sqlite.org/lang_returning.html). Sin triggers (contentless), la restricción trigger no aplica.
   - Sync FTS explícito (contentless-safe): leer los valores VIEJOS de `headers` por s3Key ANTES del upsert →
     `INSERT INTO headers_fts(headers_fts,'delete',oldRowid,oldSender,oldSubject) VALUES('delete',...)` (si existía) →
     upsert → `INSERT INTO headers_fts(rowid,sender,subject) VALUES(?,?,?)`. **NUNCA `DELETE FROM headers_fts`**
     (ilegal en contentless — verificado empíricamente en la auditoría del plan).
3. `UPDATE sync_state SET last_sk=?, synced_at=? WHERE mailbox=?` (upsert).
4. Commit.

**Idempotencia:** re-sincronizar la misma página no duplica (upsert por `s3Key`; FTS DELETE+INSERT es idempotente por
rowid). v0.5 re-sincroniza la página completa en cada `ls` (barato: ~50 filas); el `last_sk`/`sync_state`
queda registrado para un incremental real (traer solo `sk > last_sk`) como refinamiento posterior si el volumen crece.

## 6. Display y búsqueda

### 6a. Búsqueda FTS5 — subcomando CLI (`query.go` + `cmd/mail search`)

- **Subcomando nuevo `mail search <query>`** (más descubrible que un flag en `ls`; espacio para `--mailbox`,
  `--limit`, `--json`). Reusa `renderList` (6b) para la salida, así hora + s3Key aparecen también en resultados.
- Query: `SELECT h.* FROM headers_fts f JOIN headers h ON h.rowid=f.rowid WHERE headers_fts MATCH ? AND h.pk=?
  ORDER BY rank LIMIT ?` (rank más rápido que bm25() — research §2). Pesos por columna
  `bm25(headers_fts, 1.0, 2.0)` para priorizar `subject`; evaluar en smoke.
- **Escapado de input (crítico de seguridad/robustez):** envolver el query del usuario en comillas dobles y duplicar
  cada `"` interno a `""` (research §2). Neutraliza `AND`/`OR`/`NOT`/`NEAR` como operadores y caracteres especiales.
  Trade-off aceptado: deshabilita prefix `*`. Función `escapeFTS(q string) string` con test dedicado. **Compartida**
  por el CLI (6a) y la TUI (6c) — vive en `internal/cache`, no se duplica.

### 6b. Display del listado CLI (`cmd/mail/main.go renderList`)

Hoy `renderList` imprime `Date\tFrom\tSubject` con el `Date` crudo (RFC email). Cambios:
- **Fecha+hora:** `formatDate(h.Date)` → `"2006-01-02 15:04"` (ISO-like, ordenable, 24h, hora local). Resuelve el
  dolor de no distinguir correos del mismo día.
- **s3Key:** añadir el `h.S3Key` como columna en la salida tabular, para copiar directo a `mail read <s3Key>`.
  En modo `--json` ya venía (el struct completo se serializa); el cambio es solo la salida humana.
- `formatDate` reemplaza a `shortDate` (misma tolerancia multi-layout, distinto `.Format`). Función pura, testeable.

### 6c. Display + filtro de la TUI (`cmd/mail-tui/listitem.go` + `model.go`)

- **3ª línea con s3Key (dim/faint):** el `DefaultDelegate` de bubbles/list muestra 2 líneas (Title=Subject,
  Description=sender+fecha). Para la 3ª línea (s3Key) hay que **customizar el delegate**: `delegate.SetHeight(3)` +
  render propio que emita la 3ª línea con `lipgloss` faint (color tenue, menor énfasis). El s3Key queda visible y
  seleccionable con el **copy mode de tmux** (el cliente no copia). La Description pasa a usar `formatDate` (fecha+hora,
  6b) — misma función que el CLI, compartida.
- **Filtro `/` pasa de fuzzy-en-memoria a FTS5-sobre-cache:** hoy el `/` usa el filtro nativo de bubbles/list
  (`sahilm/fuzzy` sobre los items cargados, vía `FilterValue()`). En v0.5, al escribir en `/` se consulta
  `cache.Search(mailbox, query)` y se **recarga la lista** con los resultados FTS5 (mismo ranking que `mail search`).
  Al limpiar el filtro, se restaura la lista completa desde `cache.List`. **Matiz de diseño:** esto desacopla la
  búsqueda del filtro nativo → hay que interceptar el input del filtro (o desactivar `SetFilteringEnabled` nativo y
  manejar `/` propio) sin romper las invariantes de v0.4 (B-1 selección por S3Key no Index, B-2 Tab, D3 keys al filtro).
  El whole-branch review de F5 debe verificar que la interacción filtro↔selección↔debounce sigue intacta.
- Búsqueda idéntica en CLI y TUI: ambos frentes consultan el mismo FTS5 sobre el mismo cache.

## 7. Archivo, PRAGMAs, concurrencia

- **Path:** `~/.cache/erickaldama-mail/index.sqlite` (respeta `$XDG_CACHE_HOME` si está; fallback `~/.cache`).
- **Permisos:** dir `0700`, archivo **`0600` desde Go** (`os.MkdirAll` + crear/`os.Chmod` el archivo antes de que
  SQLite lo abra — SQLite no controla permisos, research §3 NO VERIFICADO). Test que verifica el modo del archivo.
- **PRAGMAs (research §3):** `journal_mode=WAL` (persistente, una vez), y **por conexión en cada Open**:
  `synchronous=NORMAL`, `busy_timeout=5000`. `db.SetMaxOpenConns(1)` — cache single-process, defensa ante
  el silencio del driver sobre concurrencia. Durabilidad ante corte de energía irrelevante (cache descartable).
- **Driver:** `_ "modernc.org/sqlite"`, `sql.Open("sqlite", dsn)` (name `"sqlite"`, NO `"sqlite3"`). CGO-free.

## 8. Dependencia

`modernc.org/sqlite@latest` — fijar en go.mod la versión que resuelva el proxy (v1.53.0 o v1.54.0, ambas
SQLite 3.53.2, FTS5 out-of-the-box sin build tag). `go mod tidy` tras añadir. Es la 1ª dep de persistencia local del
cliente; la auditoría midió **45 deps totales** (7 indirectas nuevas: modernc.org/libc, mathutil, memory,
ncruces/go-strftime, dustin/go-humanize, google/uuid, remyoudompheng/bigfft). Nota (corregida tras auditoría):
`sahilm/fuzzy` **NO es podable** — es dep transitiva de `bubbles/list` (el prompt `/` la usa). El filtro se
intercepta (no se elimina); `SetFilteringEnabled` se mantiene para la UI del prompt.

## 10. Riesgos y mitigaciones (del research NO VERIFICADO)

| Riesgo | Mitigación en el diseño |
|---|---|
| #1 firing trigger upsert (corrupción FTS silenciosa) | **Eliminado:** contentless, sin triggers, sync Go explícito |
| #2 s3Key TEXT no es rowid | **Resuelto:** rowid INTEGER explícito + s3Key UNIQUE; verificado por test de join |
| #3 concurrencia driver | `SetMaxOpenConns(1)` + `busy_timeout=5000` + WAL; fallback a reader directo |
| #4 permisos 0600 | Aplicados desde Go, verificados por test |
| #5 v1.53 vs v1.54 | Fijar la resuelta en go.mod, registrar en runbook |
| #6 FTS5 solo verificado darwin/arm64 | Test de contrato FTS5 en CI (linux) + local (darwin) |
