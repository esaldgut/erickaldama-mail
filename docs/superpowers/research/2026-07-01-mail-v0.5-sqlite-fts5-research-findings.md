## 2. Esquema FTS5 — external content

**Decisión:** external content table (`content='headers'`) — FTS5 guarda SOLO el índice, los valores se leen de
`headers` (la fuente de verdad local). NO contentless (que devuelve NULL en todo excepto rowid → no podríamos
recuperar sender/subject en resultados).

**Cita verbatim (fts5.html §4.4.3):** > "An external content FTS5 table is created by setting the content option
to the name of a table, virtual table or view ... within the same database."

**Cita verbatim (fts5.html §4.4.3, responsabilidad de sync):** > "It is still the responsibility of the user to
ensure that the contents of an external content FTS5 table are kept up to date with the content table."

### Triggers exactos (fts5.html §4.4.3, patrón oficial adaptado a headers)

```sql
CREATE VIRTUAL TABLE headers_fts USING fts5(sender, subject, content='headers', content_rowid='<rowid_col>');

CREATE TRIGGER headers_ai AFTER INSERT ON headers BEGIN
  INSERT INTO headers_fts(rowid, sender, subject) VALUES (new.<rowid_col>, new.sender, new.subject);
END;
CREATE TRIGGER headers_ad AFTER DELETE ON headers BEGIN
  INSERT INTO headers_fts(headers_fts, rowid, sender, subject) VALUES('delete', old.<rowid_col>, old.sender, old.subject);
END;
CREATE TRIGGER headers_au AFTER UPDATE ON headers BEGIN
  INSERT INTO headers_fts(headers_fts, rowid, sender, subject) VALUES('delete', old.<rowid_col>, old.sender, old.subject);
  INSERT INTO headers_fts(rowid, sender, subject) VALUES (new.<rowid_col>, new.sender, new.subject);
END;
```

**Cita verbatim (fts5.html §6.3, el 'delete' usa valores VIEJOS):** > "The values inserted into the other columns
must match the values currently stored in the table." + > "If the values 'inserted' ... are not the same as those
currently stored ... the results may be unpredictable." → el AFTER UPDATE DEBE borrar con `old.*`, no `new.*`.

### Query MATCH + ranking (fts5.html §5.1.1, §5.2)

```sql
SELECT ... FROM headers_fts WHERE headers_fts MATCH ? ORDER BY rank;  -- rank más rápido que bm25()
-- pesos por columna para priorizar subject sobre sender:
-- ORDER BY bm25(headers_fts, 1.0, 2.0)   (sender=1.0, subject=2.0)
```

**Cita verbatim (§5.2):** > "using 'rank' is faster than using bm25()." (equivale a `bm25()` sin args por defecto).

### Escapado de input de usuario (fts5.html §3.1, §3.7)

Envolver el input completo en comillas dobles `"` y duplicar cada `"` interno a `""`. Esto neutraliza `AND`/`OR`/
`NOT`/`NEAR` (case-sensitive) como operadores y protege caracteres especiales. **Trade-off documentado:** dentro de
comillas, `*` (prefix) "may not work as expected" → el envoltorio deshabilita prefix search (decisión de producto).

**Cita verbatim (§3.1):** > "Within a string, any embedded double quote characters may be escaped SQL-style - by
adding a second double-quote character."

## 3. Upsert + WAL + concurrencia + permisos

### Upsert (lang_UPSERT.html §2, §2.1)

`INSERT INTO headers (...) VALUES (...) ON CONFLICT(<uniq>) DO UPDATE SET col=excluded.col, ...` es correcto.
`excluded.` = los valores nuevos que se habrían insertado.

**Cita verbatim (§2):** > "To use the value that would have been inserted had the constraint not failed, add the
special 'excluded.' table qualifier to the column name."

**REQUISITO (§2):** el ON CONFLICT solo dispara UPSERT sobre una restricción de unicidad. > "The UPSERT processing
happens only for uniqueness constraints. A 'uniqueness constraint' is an explicit UNIQUE or PRIMARY KEY constraint
... or a unique index."

**Idempotencia por orden (§2.1, patrón oficial):** `... DO UPDATE SET ... WHERE excluded.date > headers.date` evita
sobrescribir con datos más viejos si se reprocesa un item. Útil para el sync incremental.

### WAL + PRAGMAs (wal.html §1/§3/§3.3, pragma.html)

- `PRAGMA journal_mode=WAL;` — **persistente** (una vez, sobrevive reopen). Requiere mismo host (OK, cache local).
- `PRAGMA synchronous=NORMAL;` — combo recomendado para WAL; se pierde durabilidad ante corte de energía, **irrelevante
  porque el cache es descartable** (se resincroniza desde DynamoDB). **Por conexión, re-aplicar en cada Open.**
- `PRAGMA busy_timeout=<ms>;` — mitiga SQLITE_BUSY si dos invocaciones del CLI compiten por el mismo `.db`. Sleep+retry
  hasta acumular los ms. **Por conexión.** Valor concreto (p.ej. 5000) = decisión de ingeniería, NO cita oficial.

**Cita verbatim (wal.html §3.3):** > "Unlike the other journaling modes, PRAGMA journal_mode=WAL is persistent."

**Regla de conexión:** solo WAL persiste en el archivo. `synchronous`, `busy_timeout`, `foreign_keys` son **por
conexión** → el cliente Go debe re-aplicarlos en CADA apertura.

## 4. NO VERIFICADO — huecos que F2 (spec-audit compilando) + verificación empírica deben cerrar

1. **[CRÍTICO] Firing de trigger en el path DO UPDATE de un upsert.** La doc NO confirma verbatim si
   `INSERT ... ON CONFLICT DO UPDATE` dispara el trigger `AFTER INSERT` o el `AFTER UPDATE` al caer en conflicto.
   Si el sync FTS depende del `AFTER UPDATE` (delete-viejo + re-insert) y el upsert dispara `AFTER INSERT`, el índice
   FTS se corrompe silenciosamente. **→ Verificar EMPÍRICAMENTE** (upsert contra tabla con triggers instrumentados,
   observar cuál dispara) antes de confiar el diseño. Alternativa robusta: no depender del upsert para el FTS — usar
   INSERT/UPDATE/DELETE explícitos que ejerzan los triggers de forma conocida. lang_UPSERT.html §3 solo dice: > "UPSERT
   does not intervene ... for constraints that are implemented using triggers."

2. **[CRÍTICO] `s3Key TEXT PRIMARY KEY` no es alias de rowid.** El ejemplo oficial de external content usa
   `content_rowid='a'` con `a INTEGER PRIMARY KEY` (alias de rowid). Un PK TEXT NO es el rowid. **→ Decisión de esquema:**
   `headers` debe tener una columna INTEGER rowid-alias (o usar el rowid implícito) como `content_rowid`, y `s3Key` como
   índice UNIQUE separado para el `ON CONFLICT(s3Key)`. Verificar el mapeo empíricamente.

3. **Concurrencia del driver modernc.org/sqlite.** La doc de cznic guarda silencio (solo el contrato genérico
   `database/sql` "connection used by one goroutine at a time"). `SetMaxOpenConns(1)`, WAL, busy_timeout vienen de
   terceros. **→ Apoyarse en doc de SQLite mismo + verificación empírica**; considerar `SetMaxOpenConns(1)` para el
   cache single-process como defensa.

4. **Permisos 0600 del archivo.** SQLite NO documenta control de permisos → **se aplica desde Go** (`os.OpenFile`
   con 0600 / `os.Chmod`), no desde SQLite. Verificar que crear el dir `~/.cache/erickaldama-mail/` con 0700 y el
   archivo con 0600 antes/después de que SQLite lo abra da el resultado esperado.

5. **v1.54.0 vs v1.53.0** — `go get @latest` puede resolver a cualquiera. Fijar la resuelta en go.mod y registrarla.

6. **FTS5 verificado solo en darwin/arm64** — el flag `-DSQLITE_ENABLE_FTS5` se confirmó verbatim en
   `lib/sqlite_darwin_arm64.go`. El CI corre en linux/amd64 (GitHub Actions) → confirmar el mismo flag en
   `lib/sqlite_linux_amd64.go`, o mejor: un test que haga `CREATE VIRTUAL TABLE ... USING fts5(...)` y falle si el
   módulo no está (test de contrato en ambas plataformas vía CI).
